## 3. Protocol Analysis

### 3.1 DNS (Domain Name System)
- **Port:** UDP 53 / TCP
- **Total Frames:** ~312
- **Filter:** `dns`
- **Observation:** DNS queries resolved domains for Google services, YouTube, and Microsoft.
- **Sample Queries:**
  - `accounts.youtube.com`
  - `fonts.googleapis.com`
  - `play.google.com`
  - `edge.microsoft.com`

### 3.2 TCP (Transmission Control Protocol)
- **Total Frames:** ~1566
- **Filter:** `tcp`
- **Observation:** TCP three-way handshake (SYN → SYN-ACK → ACK) visible for all new connections. Used as transport for TLS, HTTP, and DNS-over-TCP.

### 3.3 TLS / HTTPS
- **Port:** TCP 443
- **Total Frames:** ~107
- **Filter:** `tls`
- **Observation:** Encrypted web traffic. Client Hello and Server Hello handshakes visible. Payload is fully encrypted — only metadata (IPs, ports, SNI) observable.

### 3.4 QUIC
- **Transport:** UDP
- **Total Frames:** 2265
- **Filter:** `quic`
- **Observation:** QUIC is Google's modern transport protocol used by Chrome for YouTube and Google services. Faster than TCP+TLS with built-in encryption. Majority of traffic in this capture.

### 3.5 ICMP (Internet Control Message Protocol)
- **Total Frames:** 8
- **Filter:** `icmp`
- **Observation:** Generated by `ping 8.8.8.8`.
  - Type 8 = Echo Request (host → 8.8.8.8)
  - Type 0 = Echo Reply (8.8.8.8 → host)

### 3.6 ARP (Address Resolution Protocol)
- **Total Frames:** 4
- **Filter:** `arp`
- **Observation:** Broadcast frames for IP-to-MAC resolution on the local network subnet.

## 5. Key Findings

- **QUIC dominates traffic** — Most Google/YouTube traffic uses QUIC over UDP, not traditional TCP+TLS.
- **DNS is plaintext** — All queried domain names are visible (privacy risk without DoH/DoT).
- **TLS hides payload** — HTTPS connections show only handshake metadata; content is unreadable.
- **ICMP confirms connectivity** — Ping to 8.8.8.8 confirmed successful with Echo Reply responses.
- **ARP visible** — Local network MAC resolution captured via broadcast frames.

## 6. Interview Question Answers

**Q1. What is Wireshark used for?**
Wireshark is a network protocol analyzer used to capture and inspect packets on a network interface in real time for troubleshooting, security analysis, and protocol learning.

**Q2. What is a packet?**
A packet is a unit of data transmitted over a network, containing a header (source/destination IPs, protocol) and a payload (actual data).

**Q3. How to filter packets in Wireshark?**
Using display filters in the filter bar. Examples: `dns`, `tcp`, `icmp`, `ip.addr == 8.8.8.8`, `tcp.port == 443`.

**Q4. What is the difference between TCP and UDP?**
TCP is connection-oriented with guaranteed, ordered delivery using handshakes and acknowledgments. UDP is connectionless and faster but with no delivery guarantee — used for DNS, QUIC, video streaming.

**Q5. What is a DNS query packet?**
A DNS query is sent by a client to a DNS server to resolve a domain name to an IP address. It uses UDP port 53 and is visible in plaintext.

**Q6. How can packet capture help in troubleshooting?**
It allows engineers to see exact traffic — identify failed connections, unusual traffic patterns, misconfigured protocols, latency issues, or network attacks.

**Q7. What is a protocol?**
A protocol is a set of rules governing how data is formatted and transmitted between devices (e.g., TCP, DNS, QUIC, HTTP).

**Q8. Can Wireshark decrypt encrypted traffic?**
Not by default. TLS/QUIC traffic can only be decrypted if session keys are provided via an SSL key log file. Without keys, only metadata is visible.
